# CTF-LAB

我的个人 CTF 训练仓库，用于记录刷题过程、沉淀解题方法、积累工具脚本，并在多台设备之间同步训练环境。

当前阶段以 **picoCTF 基础入门** 和 **BUUCTF 基础训练** 为主，目标不是单纯“刷出多少题”，而是逐步建立起完整的：

- Web
- Crypto
- Forensics
- Pwn
- Rev

五大方向的基础能力、分析习惯、复盘能力与个人知识库。

---

## 仓库目标

这个仓库承担四个核心作用：

### 1. 题目归档
按 **平台 / 方向 / 题目名称** 归档每一道题，保证训练内容长期可追踪。

### 2. 过程记录
记录每道题的：

- 题面关键信息
- 初步判断
- 尝试路径
- 关键命令
- 最终解法
- 复盘总结

### 3. 知识沉淀
把题目中抽象出来的通用方法整理进 `notes/`，形成自己的知识体系。

### 4. 工具积累
把常用脚本、字典、样本、自动化小工具沉淀到 `tools/` 中，逐步建立个人武器库。

## 使用规则

### 1. 一题一个文件夹

每道题都必须放进独立目录，避免附件、笔记、脚本混乱。

例如：

```text
picoctf/forensics/riddle-registry/
```

### 2. 每题至少包含这些内容

```text
题目目录/
├── 原始附件
├── solve.md
├── notes.txt
├── extracted/
└── screenshots/
```

说明：

* `solve.md`：正式题解 / Writeup
* `notes.txt`：临时记录、草稿命令、失败尝试、碎片思路
* `extracted/`：提取出的文件、中间产物
* `screenshots/`：关键截图

### 3. 题解和方法论必须分开

* `solve.md` 记录：**这道题怎么做**
* `notes/` 记录：**这类题一般怎么做**

题解是案例，`notes/` 是知识树。

### 4. 每做完一题都必须复盘

不允许“做完就扔”。
每题至少回答：

* 这题真正考什么？
* 题面哪个词是提示？
* 我一开始卡在哪里？
* 下次再遇到类似题，第一步该做什么？

### 5. 每做完一题都进行 Git 提交

保持训练过程可追踪、可同步。

# 做题方法（核心作战流程）

这一部分是本仓库最重要的规则。
以后做任何一道题，原则上都按这个流程走。

---

## 总流程

### Step 1. 读题

先完整读题，不要急着乱点乱试。

要做的事：

* 看题目标题
* 看题目分类
* 看题面描述
* 看附件类型
* 看提示（Hints）

要问自己的问题：

* 这是哪个方向？
* 题面有没有强提示词？
* 附件是什么格式？
* 这题更像“考工具使用”还是“考漏洞点”？

---

## Step 2. 判题型

先判断这题属于哪一类，再决定工具，不要无脑乱试。

常见分类：

* Web
* Crypto
* Forensics
* Pwn
* Rev
* General Skills / Misc

例子：

* 给你网页链接，多半是 Web
* 给你加密字符串，多半是 Crypto
* 给你图片、pdf、pcap、压缩包，多半是 Forensics
* 给你 ELF 可执行文件，多半是 Pwn 或 Rev

---

## Step 3. 提取关键词

把题面里的关键词记下来，因为很多题提示其实已经写在题面里了。

例如：

* `metadata`
* `cookie`
* `session`
* `upload`
* `robots.txt`
* `xor`
* `base64`
* `overflow`
* `checksec`
* `assembly`

建议在 `notes.txt` 里先写：

```text
关键词：
- 
- 
- 
```

---

## Step 4. 写下初步猜测

先做一个低成本、可修正的猜测，不要求第一次就对。

例如：

* 这题可能是 PDF 元数据题
* 这题可能是简单 SQL 注入
* 这题可能是查看 cookies / headers
* 这题可能是栈溢出入门
* 这题可能是字符串混淆 / 简单逆向

建议写在 `notes.txt`：

```text
初步判断：
- 
可能工具：
- 
```

---

## Step 5. 先做低成本检查

这是非常重要的习惯。

原则：

**先做信息量最大、成本最低的检查，再深入挖掘。**

不要一上来就：

* 爆破
* 硬调试
* 乱写脚本
* 乱猜 payload

先做最基础的信息收集。

---

## Step 6. 根据结果缩小范围

看输出，缩小方向，再决定下一步，而不是一直盲打。

比如：

* `file` 发现其实是 zip，不是 png
* `strings` 发现可疑字符串
* `exiftool` 发现 metadata 中有 flag
* Burp 抓包发现参数可控
* `checksec` 发现没开 canary

这些都属于“缩小范围”。

---

## Step 7. 记录关键现象和关键命令

做题过程中一定要留痕。

至少记：

* 我输入了什么命令
* 输出里有什么异常
* 哪一步让我确定了方向
* 哪个尝试失败了，为什么失败

`notes.txt` 可以乱，`solve.md` 要清晰。

---

## Step 8. 拿到 flag

拿到 flag 不代表结束。

真正完整的一题，应该做到：

* 知道为什么是这个 flag
* 知道为什么这套方法有效
* 知道这题属于什么模板

---

## Step 9. 整理正式题解

把草稿整理成 `solve.md`。

正式题解至少包括：

* 题面信息
* 分析路径
* 关键命令
* 最终结果
* 复盘

---

## Step 10. 抽象成方法论

把通用部分提炼进 `notes/`。

例如这题学到的是：

* PDF 题起手先看 metadata
* 图片题先看 strings / exif / binwalk
* Web 题先看请求与响应
* Pwn 题先 `checksec`

那这些应该进入 `notes/`，而不是只留在某一道题里。

---

## Step 11. Git 提交并同步

每做完一题：

```bash
git add .
git commit -m "solve picoctf forensics riddle-registry"
git push
```

# 各方向起手方法

这一部分用于“刚看到题时我该先做什么”。

---

## 1. Forensics 起手方法

适用对象：

* 图片
* PDF
* 音频
* 视频
* 压缩包
* pcap
* office 文档
* 二进制附件

优先顺序：

### 第一层：基础识别

```bash
file <filename>
strings <filename> | less
xxd <filename> | head
```

目的：

* 识别真实文件类型
* 看是否藏有明文信息
* 看文件头是否异常

### 第二层：元数据检查

```bash
exiftool <filename>
pdfinfo <filename>
```

目的：

* 检查 metadata
* 看作者、标题、备注、自定义字段

### 第三层：内容提取

```bash
binwalk <filename>
foremost <filename>
```

目的：

* 看是否有嵌套文件、隐藏文件、追加数据

### 第四层：特殊格式专项分析

例如：

* 图片：`zsteg`、`steghide`
* 压缩包：密码爆破、伪加密
* pcap：Wireshark、tshark、提取 HTTP/FTP 流量
* PDF：对象结构、附件、脚本、metadata

核心习惯：

* 先看真实类型
* 先看字符串
* 先看 metadata
* 再做深挖

---

## 2. Web 起手方法

适用对象：

* 网页题
* 登录框
* 参数题
* 文件上传
* API
* cookie / session 题

优先顺序：

### 第一层：直接观察

* 打开网页
* 点页面上所有能点的地方
* 看 URL 参数变化
* 看输入框、按钮、返回信息

### 第二层：看源码

* `F12`
* 查看页面源码
* 看 JS
* 看注释
* 看隐藏字段

### 第三层：抓包

使用 Burp Suite：

* 请求头
* 响应头
* Cookie
* Session
* 参数值
* 接口路径
* 重定向行为

### 第四层：测试常见入口

常见检查点：

* URL 参数
* POST 参数
* Cookie
* Header
* 文件上传点
* 目录扫描
* robots.txt
* .git
* 备份文件
* debug 接口

### 第五层：判断漏洞方向

常见方向：

* SQL 注入
* 命令执行
* 文件包含
* 文件上传
* 身份认证绕过
* 权限控制缺陷
* 反序列化
* SSTI
* XSS

核心习惯：

* 先看请求和响应
* 先确认“哪里可控”
* 再决定 payload，不要盲注

---

## 3. Crypto 起手方法

适用对象：

* 密文
* 编码串
* 哈希串
* 古典密码
* XOR
* RSA
* 数论题

优先顺序：

### 第一层：识别格式

先看它像什么：

* Base64？
* Hex？
* URL 编码？
* 哈希？
* 二进制字符串？
* 栅栏 / 凯撒 / 维吉尼亚？

### 第二层：做简单变换

先尝试：

* 编解码
* 转换进制
* 大小写/分组
* 异或猜测
* 频率分析

### 第三层：观察题面暗示

题面可能提示：

* shift
* Caesar
* XOR
* key
* prime
* RSA
* modulus

### 第四层：写脚本验证

Crypto 很多题不适合手算，及时用 Python 验证。

核心习惯：

* 先识别是什么，不要一上来乱解
* 先试最常见编码和简单密码
* 不懂就写脚本枚举验证

---

## 4. Pwn 起手方法

适用对象：

* ELF 二进制
* 交互式程序
* 给源码或不给源码的本地程序

优先顺序：

### 第一层：识别程序信息

```bash
file ./chall
checksec ./chall
```

看：

* 32 位 / 64 位
* 是否动态链接
* NX
* Canary
* PIE
* RELRO

### 第二层：运行程序

```bash
./chall
```

观察：

* 有哪些输入点
* 有哪些输出点
* 程序功能是什么
* 哪里可能有异常

### 第三层：看源码 / 反汇编

如果有源码，先看源码。
如果没源码：

* strings
* Ghidra / IDA
* objdump
* gdb

### 第四层：找漏洞类型

常见：

* 栈溢出
* 格式化字符串
* 堆漏洞
* 整数溢出
* UAF
* ret2win
* ret2libc

核心习惯：

* 起手一定 `checksec`
* 先理解程序逻辑
* 不要上来就套 exp 模板

---

## 5. Rev 起手方法

适用对象：

* ELF / PE / APK
* 混淆程序
* 校验逻辑
* 输入正确 flag 的逆向题

优先顺序：

### 第一层：基础信息

```bash
file <binary>
strings <binary> | less
```

### 第二层：动态观察

```bash
ltrace ./chall
strace ./chall
```

看程序做了什么系统调用、调用了什么库函数。

### 第三层：静态分析

工具：

* Ghidra
* IDA
* Cutter
* objdump

关注点：

* main 函数
* 输入处理
* 比较逻辑
* 加密 / 解密函数
* 关键字符串引用

### 第四层：还原逻辑

目标不是“看懂所有汇编”，而是：

* 找到校验点
* 找到变换过程
* 反推出输入

核心习惯：

* 先 strings
* 再找 main 和关键分支
* 先看“比较发生在哪里”

# AI 辅助做题原则

AI 可以辅助，但不能替代你的判断。
本仓库默认遵守以下规则：

## 1. 先自己判型，再问 AI

先自己回答：

* 这题是什么方向？
* 第一批工具是什么？
* 我准备先做什么？

## 2. 问 AI 要问“方法”，不要只问“答案”

优先问：

* 这题像什么类型？
* 下一步该检查什么？
* 为什么优先这个工具？
* 这个输出说明了什么？

## 3. 卡住时向 AI 汇报“现象”

错误问法：

* 这题怎么做？

更好的问法：

* 我已经执行了 `pdfinfo` 和 `exiftool`，发现这些字段，下一步该往哪个方向看？

## 4. 最终要自己复盘

即使 AI 给了提示，最后也要自己写明：

* 这题为什么这么解
* 这一类题的通用方法是什么

`notes/` 存放的是专题方法，不是某一道题的完整解法。
